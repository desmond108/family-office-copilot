Turning ad-hoc client asks into a proposal the AI can write

How we capture varied, in-passing client instructions — “buy in tranches,” “gold below $4,000,” “low fees, good quality” — and pass them straight to the AI Model, without ever letting it invent a number.

The question

“How do we put these asks into the UI — ‘buying in tranches,’ ‘interest-rate expectations,’ ‘low fees, good quality’? Just another free-text box? They can be very ad-hoc and very varied.” — the question we're answering

Short answer (v10): yes — a free-text box, and we pass the client's words verbatim to the AI Model, together with everything else the copilot knows, in one self-contained prompt. The AI analyses that material and writes the proposal.

The v10 shift: pass-through, not pre-digest

This supersedes v6–v8. Earlier versions sorted the free text into typed items (Sort → review → confirm), tagged an enforcement tier, and let a price trigger gate a trade. v10 removes that pipeline: modern AI models read the variety directly, so the copilot hands them the raw inputs and makes the prompt itself the product surface. Why the prompt, not a review table

The prompt is the product Transparent

  • Shown on the Proposal page — editable, copyable, downloadable.
  • Paste it into any AI model to reproduce & compare.
  • The client reads exactly what the system asks the AI to do.

Numbers stay deterministic Trustworthy

  • Every figure comes from the engine (the FACTS block).
  • The AI writes prose only — quoting, never altering.
  • Qualitative claims from docs are context, not verified.

The rule that shapes everything. The system never invents a number. The AI may quote the FACTS figures but is instructed never to compute, estimate, re-round or fabricate one. Because the model now reads the client's documents and instructions directly, the deck's commentary note discloses that qualitative statements are supplied as context and not independently verified.

The flow, end to end

Four phases — the analyst sets the policy and pastes the inputs; the copilot computes the numbers and assembles the prompt; the AI Model writes the narrative.

Capture

Paste tactical text (verbatim) + upload documents; set the policy.

Compute

The engine parses the book and computes the deterministic FACTS.

Assemble

The copilot builds one self-contained prompt — visible & editable.

Generate

The AI Model writes the narrative — in-app, or copy into any model.

→ a proposal deck (PPTX / PDF): AI prose around deterministic tables

What's built, and the trade-offs

Shipped — live on Streamlit

  • A Tactical instructions free-text box — no sorting, no review table, no enforcement tiers.
  • The full self-contained prompt on the Proposal page: editable, copyable, downloadable.
  • Generate with the AI Model (live API when a key is set), or copy the prompt into any AI model.
  • Allocation, drift, rebalance & suitability stay computed by the engine and unchanged by the model.
  • Model-agnostic UI — every label reads “AI Model”. Works with or without an API key.

Trade-offs vs. v6–v8

  • Lost: the monitoring watchlist, the per-item review/confirm gate, and the one enforced price trigger that could gate a trade.
  • If a binding conditional rule is needed again, it returns as a deterministic check in the engine — never as model output.
  • Gained: a single, legible, portable prompt the client can inspect and test across models; far less UI friction; fused/ambiguous asks handled directly.

Bottom line

The free-text box stays as the capture surface. v10 stops pre-digesting the client's words and instead shows — in one transparent, model-agnostic prompt — exactly what the AI is asked to do, while every number remains computed deterministically by the engine.
