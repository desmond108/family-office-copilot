Turning ad-hoc client asks into things the copilot can act on

How we capture varied, in-passing client instructions — “buy in tranches,” “gold below $4,000,” “low fees, good quality” — without ever letting the model invent a number.

The question

“How do we put these asks into the UI — ‘buying in tranches,’ ‘interest-rate expectations,’ ‘low fees, good quality’? Just another free-text box? They can be very ad-hoc and very varied.” — the question we're answering

Short answer: keep a free-text box — but treat it as the way in, not the way it's stored. A single box is the weakest option for this product, and the fix is small because it reuses what we already built.

Why not just one free-text box

One box Loses what matters

  • Everything becomes prose the model paraphrases.
  • “Gold below $4,000” is never watched — just restated.
  • Fused asks stay fused (see below).

Typed slots Keeps it actionable

  • Each ask lands where it can be acted on.
  • Levels are copied from the client, never invented.
  • A human confirms before anything counts.
“buy the bond fund in tranches — but what about rate hikes?”
one line, two asks →  Execution style + Open question
§

The rule that shapes everything. The system never invents a number. The model only sorts the client's words and copies the levels they actually stated — if a level isn't given, the field stays empty. That's the difference between a note and a monitorable rule.

The pattern — reused from statement ingestion

The same capture → sort → review → confirm gate we built for reading PDF statements in v5. It fits the discipline instead of fighting it.

Free text

Client's asks, typed in plain language.

Sort into types

The model classifies each ask — copying levels, inventing none.

Review & edit

Keep / ignore, fix the type or level, add a note.

Confirm

Only kept, resolved items count.

→ drives the proposal as guidance, and triggers become a monitoring watchlist

✓

In the review table, the analyst is in control of every row. Keep (a checkbox) ignores an item without deleting it; Note records what to raise with the client; and anything the model couldn't act on is typed Needs clarification — shown in a ⚠️ callout and held out of the proposal until it's re-typed or removed. Unclear input never reaches the deck.

Target allocations: propose, don't auto-fill

When the instructions state target weights, the copilot aggregates them per sleeve — Nasdaq 20% + S&P 30% → equity 50% — and shows a 📥 Proposed allocation panel. Nothing is written silently:

Apply Fill the sleeves

  • Writes the proposed weights into the allocation targets.
  • Still fully adjustable afterward.

Ignore Keep your own

  • Dismisses the panel and the nudge.
  • A different later set of client weights re-surfaces it.

The client's stated weights drive the analysis only on an explicit click. Limits (band tolerance, liquidity, FX, position caps) stay fully manual.

What's built, and what's next

Shipped — live on Streamlit

  • A Tactical instructions box → Sort → review table (Keep · Note · type · level, ⚠️ items held out) → Confirm.
  • Propose → Apply / Ignore allocation targets when the client stated weights.
  • Works with or without an API key (a keyword pass runs the keyless demo).
  • Verified on the client's message: allocations mapped per sleeve (=100%), triggers captured with their levels, the tranche/rate-hike ask split apart.

Honest limits / next

  • The watchlist is captured, not yet live-monitored — wiring it to price alerts is the next step, and the highest-value one.
  • Selection criteria are captured as guidance, not yet a live product screen.
  • The keyword pass is a first draft; that's exactly why a human confirms.

Bottom line

Yes to a free-text box — as the capture surface, not the format. Sorting ad-hoc guidance into typed, human-confirmed items turns it into things the copilot can act on and monitor — proposing allocations, watching triggers, flagging what's unclear — without ever letting the model invent a number.
